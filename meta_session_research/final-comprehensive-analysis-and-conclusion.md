# Final Comprehensive Analysis And Conclusion

Date: 2026-05-01

Status: final conclusion for this meta-session research pass. Not an
implementation plan.

## Executive Conclusion

Azoth should not remain a single always-on full harness.

The evidence supports a profile-split operating model:

- `azoth-lite` as the default posture for ordinary work, meta work, focused
  verification, and pre-action judgment;
- `azoth-full` as governed mode for packaging, closeout, final delivery,
  governed-state mutation, release, adapter parity, autonomous continuation, and
  other high-audit work;
- `stock-lite` as a continuing baseline and acceptable mode for simple
  read-only/status/test tasks where audit requirements are low;
- `meta-harness-experimental` as the long-term architecture target: one
  strategic brain, explicit side-effectful hands, durable session/event log,
  context views, progressive-disclosure skills, permission gates, and trace
  grading.

The shortest form:

Azoth should become lighter by default, stricter on escalation, and more
explicit about when governed machinery is actually earning its keep.

## Why This Is The Right Conclusion

### Full Azoth Is Too Heavy As A Universal Default

The meta-artifact correction case and everyday verification case both show that
full Azoth can pull work toward native artifact shapes, route ceremony, and
process overhead when the task mostly needs intent fidelity, focused tests, or a
small trace.

This matters because GPT-5.5-class models are capable enough that the harness
should not spend the model's attention navigating scaffolding unless the
scaffolding is buying safety, auditability, or recovery.

### Full Azoth Is Still Valuable

The hydration boundary and governed packaging cases show that deterministic
structure is not mere ceremony. Hydration had a real historical bypass. Case 7
showed that when the task becomes "finish/package safely" across governed state,
`azoth-full` gives useful language and constraints:

- human and agent gates;
- packaging and final-delivery discipline;
- governed artifact disposition;
- verification evidence;
- closeout/handoff treatment;
- audit trail preservation.

The right move is not to delete full Azoth. The right move is to stop making it
the default shape for every task.

### Stock-Lite Is Useful But Insufficient

Stock-lite performed well in fresh packet runs because the fixtures were clear
and the tasks were answer-only. It is fast and low-friction. It should remain a
baseline and may be acceptable for low-risk read-only work.

But it lacks durable gate mechanics, escalation structure, and reusable
traceability. It should not be the only operating mode for a repo whose failure
modes include governed state, packaging, roadmap truth, closeout, and
autonomous continuation.

### Azoth-Lite Is The Best Current Default Candidate

Across the research, `azoth-lite` preserved the useful primitives without
loading the full operating system:

- side-effect classification;
- dirty-worktree respect;
- explicit stop state;
- no fake finality;
- minimal trace;
- skill loading by trigger;
- escalation when governed state appears.

The manual `azoth-lite` trial also showed that the profile can remain small:
context view, side-effect class, escalation checklist, trace note, stop state.

The caveat is important: `azoth-lite` is credible as a runbook, not yet as a
productized runtime.

### Meta-Harness Is The Target Architecture

The external evidence and internal pilots converge on the same shape:

- model as strategic brain;
- tools and scripts as explicit hands;
- session/event log as durable truth;
- context views instead of always-loaded doctrine;
- skills as progressive disclosure;
- runtime guards and permission gates instead of prompt walls;
- evals and trace grading before architecture changes.

That is the long-term architecture Azoth should move toward if migration is
approved later.

## What To Preserve

Preserve these as trust primitives:

- side-effect classification;
- human gates for governed, destructive, release, dependency, kernel, and final
  delivery actions;
- dirty-worktree awareness;
- explicit stop states;
- traceability;
- run evidence for governed work;
- reflection/audit trail;
- focused verification;
- progressive skills;
- adapter parity discipline when cross-surface changes occur.

## What To Shrink

Shrink these as defaults:

- always-loaded command and pipeline doctrine;
- roadmap-first routing for tasks that are not roadmap work;
- native proposal/validator creation during meta-analysis;
- closeout rituals for answer-only or focused verification tasks;
- mandatory multi-agent/stage structure for low-risk work;
- broad context loading before the task demonstrates risk.

## What To Keep As Governed Mode

Keep `azoth-full` for:

- packaging and final delivery;
- closeout;
- `.azoth` roadmap/backlog/initiative/run-ledger/memory mutation;
- kernel and governance work;
- release, deploy, publish, merge;
- adapter parity changes;
- autonomous continuation;
- high-audit multi-stage delivery;
- tasks where independent review or fresh context is materially useful.

## Recommended Operating Model

Start every task in `azoth-lite` unless the user explicitly requests full
governance or the task immediately triggers escalation.

Escalate to `azoth-full` when:

- governed state would change;
- final delivery, packaging, closeout, release, publish, or merge is requested;
- kernel/governance/dependency/external/destructive actions appear;
- autonomous continuation is requested;
- multi-stage independent review is genuinely needed.

Use `stock-lite` as:

- a baseline for evaluation;
- a low-risk read-only/status mode;
- a sanity check against harness overreach.

Design toward `meta-harness-experimental` as:

- the eventual substrate;
- not the current runtime.

## Confidence

Strategic conclusion confidence: medium-high.

Implementation-readiness confidence: low.

Reason:

The strategic pattern is consistent across external evidence, internal
cartography, friction records, same-session pilots, fresh packet runs, and a
manual `azoth-lite` trial. However, the research still lacks:

- a real narrow code edit comparison;
- tool-enabled isolated-worktree packaging action;
- implemented `azoth-lite` machinery;
- measured user burden across repeated real sessions.

## Final Recommendation

Adopt the profile split as the research conclusion:

1. `azoth-lite` default.
2. `azoth-full` governed mode.
3. `stock-lite` baseline/simple read-only mode.
4. `meta-harness-experimental` long-term architecture target.

Do not create implementation artifacts yet unless the human explicitly starts a
new migration-planning phase.

The next phase, if approved, should be design planning for how to operationalize
the split without recreating full Azoth under a lighter name.

## Closing Judgment

Azoth's original "be water" philosophy still looks right. The current system
became too solid in places. The answer is not to evaporate it into nothing; it
is to restore phase change.

Default to flow. Freeze only where trust requires structure.
